# Mini App Testing & Local Development

## Problem

The Mini App HTTP layer (`internal/bot/http.go`) has zero test coverage. Locally, the web-app cannot be opened in a browser because the frontend crashes without `window.Telegram.WebApp`. Testing the full flow in Telegram requires a public HTTPS URL.

## Design

Two complementary workstreams: automated tests for CI, and local dev experience for manual testing.

### 1. Minimal Production Refactoring

**Extract bot token from `validateTelegramInitData`.**

Current signature ties auth to the concrete `bot.Bot` type:

```go
func (hs *HTTPServer) validateTelegramInitData(initData string) (int64, error)
```

New signature â€” a standalone function accepting token as parameter:

```go
func validateTelegramInitData(initData string, botToken string) (int64, error)
```

The `authMiddleware` method calls it with `hs.bot.api.Token()`.

The `authMiddleware` method calls it with `hs.bot.api.Token()`. Tests call `validateTelegramInitData(data, "test-token")` directly â€” no struct changes needed for auth unit tests.

For middleware-level tests in webhook mode, the test helper needs a way to provide the token without a real `bot.Bot` API client. Add a `botToken` string field to `HTTPServer`:

```go
type HTTPServer struct {
    bot         *Bot
    webhookMode bool
    botToken    string  // set from api.Token() at construction time
}
```

`authMiddleware` uses `hs.botToken`. The `RegisterRoutes` constructor (or wherever `HTTPServer` is created) sets `botToken` from `hs.bot.api.Token()`. In tests, set it directly to a known value.

### 2. Automated Tests

#### `internal/bot/auth_test.go` â€” auth validation

Test helper generates valid Telegram initData:

```go
func generateTestInitData(t *testing.T, token string, userID int64, authDate time.Time) string
```

Builds URL-encoded string with `user`, `auth_date`, sorted keys, HMAC-SHA256 hash â€” the same algorithm Telegram uses.

Test cases:
- Valid initData returns correct userID
- Tampered hash returns error
- Expired auth_date (>24h) returns error
- Empty string returns error
- Missing `user` field returns error

#### `internal/bot/http_test.go` â€” endpoint tests

Test helper creates an `HTTPServer` in polling mode with mock storage:

```go
func newTestHTTPServer(t *testing.T) *HTTPServer {
    mockDB := stubs.NewMockDB()
    b := &Bot{
        db:                 mockDB,
        allowedUsers:       map[int64]bool{123: true},
        states:             make(map[int64]*ConversationState),
        logger:             zap.NewNop(),
        notificationChatID: 0, // must be 0 â€” non-zero would call bot.api (nil) and panic
    }
    return &HTTPServer{bot: b, webhookMode: false}
}
```

For webhook-mode auth tests, a separate helper adds `botToken` and `webhookMode: true`:

```go
func newTestHTTPServerWebhook(t *testing.T) *HTTPServer {
    hs := newTestHTTPServer(t)
    hs.webhookMode = true
    hs.botToken = "test-bot-token"
    return hs
}
```

Uses `httptest.NewRecorder` + direct handler calls (no need to start a server).

**GET /api/books:**
- Returns 200 with JSON array of readable books
- Response deserializes to `[]models.Book`

**GET /api/participants:**
- Returns 200 with JSON array of participants
- Contains both children and parents

**POST /api/events:**
- Valid JSON body â†’ 201, event exists in mock storage
- Empty body â†’ 400
- Invalid JSON â†’ 400
- Missing required fields â†’ 400

**GET /web-app:**
- Returns 200 with HTML content

#### Auth middleware tests (webhook mode)

Separate tests using `newTestHTTPServerWebhook` with `webhookMode: true` and `botToken` set:
- No Authorization header â†’ 401
- Invalid initData â†’ 401
- Valid initData (generated by helper) with allowed userID â†’ handler called
- Valid initData with disallowed userID â†’ 401

### 3. Telegram SDK Mock Shim in `index.html`

Add at the top of the `<script>` block, before any `tg.*` usage:

```javascript
if (!window.Telegram?.WebApp?.initData) {
    console.warn("[DEV] Telegram WebApp SDK not found, using mock");
    window.Telegram = { WebApp: {
        initData: "",
        initDataUnsafe: { user: { id: 0, first_name: "Dev" } },
        themeParams: {},
        colorScheme: "light",
        expand: function() {},
        ready: function() {},
        close: function() { window.close(); },
        MainButton: { show: function(){}, hide: function(){}, setText: function(){} },
    }};
}
```

**Safety:** The shim only activates when `window.Telegram.WebApp.initData` is falsy. Inside Telegram, `initData` is always a non-empty string â€” the shim does not fire under normal conditions. Edge case: if the Telegram CDN script fails to load, the shim would activate even inside Telegram, but this is safe â€” webhook-mode auth would reject the empty initData with a 401. The backend already skips auth in polling mode, so empty `initData` from the shim works for local development.

### 4. Local Development Workflow

**Browser (daily development):**
1. `make run-dev` â€” starts bot in polling mode with ClickHouse testcontainer
2. Open `http://localhost:8081/web-app` in browser
3. Shim provides mock Telegram SDK, auth is skipped
4. Full UI interaction: pick date, search books, select participant, submit event

**Telegram (integration check):**
1. `make run-dev` in one terminal
2. `cloudflared tunnel --url http://localhost:8081` in another â€” get public HTTPS URL
3. In BotFather: set Mini App URL to the tunnel URL (or use inline keyboard with the URL)
4. Open Mini App in Telegram â€” real SDK, real initData, real theme

## Out of Scope

- E2E tests with real Telegram API
- Testing notification sending (requires mocking Telegram send API â€” separate task)
- Frontend JS unit tests (minimal JS logic, not worth the tooling)
- Stable tunnel URLs (paid ngrok feature, not needed for occasional testing)
